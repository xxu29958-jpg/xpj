# 后端错误码 → Android 用户文案映射表

本表用于统一后端 `AppError` 错误码到 Android 灰度用户可见文案，避免各页面各说各的。

Android Repository 优先解析后端的 `error` 字段；如果是本表中的已知错误码，统一使用本表文案。未知错误码才回退到后端 `message` 字段。

## 映射表

### invalid_token

| 字段 | 内容 |
|:---|:---|
| 后端含义 | Session token 无效或已过期。可能原因：token 被撤销、设备被移除、Authorization Header 格式不对。 |
| 用户文案 | 绑定已失效，请重新绑定账本。 |
| 用户下一步动作 | 1. 向服务拥有者申请新的绑定码重新绑定。<br>2. 确认设备未被服务拥有者移除。<br>3. 在设置页"清除服务器绑定"后重新绑定。 |
| 是否可重试 | 否（需重新绑定后才可重试）。 |
| 是否敏感 | 否（不暴露具体 Token 值）。 |
| 是否阻断当前操作 | 是（所有受保护接口都会返回 401/403）。 |

| 是否可重试 | 否（必须使用新凭证）。 |
| 是否敏感 | 否。 |
| 是否阻断当前操作 | 是。 |

### invalid_pairing_code

| 字段 | 内容 |
|:---|:---|
| 后端含义 | 绑定码无效、不存在、格式不对、已过期、已被其它设备使用、已被清理，或同一来源短时间内失败次数过多被限流。为避免暴露绑定码是否存在，以上情况统一返回此码（HTTP 401），不再细分过期/已用。 |
| 用户文案 | 绑定码无效。 |
| 用户下一步动作 | 1. 确认输入的是 8 位数字。<br>2. 向服务拥有者重新获取有效绑定码。<br>3. 如果连续输错很多次，等待一段时间后再试。 |
| 是否可重试 | 是（确认绑定码正确后可重试）。 |
| 是否敏感 | 否。 |
| 是否阻断当前操作 | 是（绑定失败）。 |

> 历史上区分过 `pairing_code_expired`（410）/ `pairing_code_used`（409），v1.1
> 起合并进 `invalid_pairing_code`（401），避免暴露绑定码是否存在。客户端不再
> 需要为这两个码单独映射文案。

### idempotency_key_required

> ADR-0042：以下三个幂等键错误码属于协议层。经 outbox 重放的写请求总会自动带上 `Idempotency-Key`，正常使用不会触发；列在此处供排查与兜底文案。用户文案不出现"幂等键"等术语（ENGINEERING_RULES §10）。覆盖路由（非"仅 PATCH"）：expense 的 patch / confirm / reject / mark-not-duplicate / ocr-retry / recognize-text / acknowledge-mismatch / replace-items / replace-splits，以及 rule 的 update+delete、merchant alias 的 update+delete（完整清单见 `docs/architecture/API.md` 的"乐观并发与请求幂等键"）。

| 字段 | 内容 |
|:---|:---|
| 后端含义 | 上述 outbox 路由的写请求缺少 `Idempotency-Key` 头。客户端 outbox 会在意图发生时生成 UUID 并随直发与重放一起带上，缺失说明请求未经正常路径。 |
| 用户文案 | 操作未能完成，请重试。 |
| 用户下一步动作 | 1. 重试一次。<br>2. 若反复出现，更新 App。 |
| 是否可重试 | 是。 |
| 是否敏感 | 否。 |
| 是否阻断当前操作 | 是（本次请求被拒绝，状态码 422）。 |

---

## HTTP 状态码兜底映射（无 error 字段时）

当后端返回非 200 但未携带可解析的 `{"error":"..."}` 结构时，Android Repository 按状态码兜底：

| HTTP 状态码 | 用户文案 | 是否可重试 | 是否阻断 |
|:---:|:---|:---:|:---:|
| 401 / 403 | 绑定已失效，请重新绑定账本。 | 否（需重新绑定） | 是 |
| 404 | 账单不存在。 | 否 | 是 |
| 413 | 上传文件超过大小限制。 | 是（压缩后） | 是 |
| 502 / 503 / 504 | 小票夹服务暂时没有响应，请稍后再试。 | 是 | 是 |
| 其他 | 连接出错（$statusCode），请稍后再试。 | 视情况 | 是 |

## 网络层错误（无 HTTP 响应时）

当请求未能到达后端（IOException）时，Android 使用 `NetworkErrorMapper` 的 `userNetworkMessage`：

| 异常类型 | 用户文案 | 是否可重试 | 是否阻断 |
|:---|:---|:---:|:---:|
| SSLHandshakeException | 当前网络或 VPN 可能拦截了小票夹连接，请稍后重试或切换网络。 | 是 | 是 |
| SocketTimeoutException | 网络响应有点慢，请稍后重试。 | 是 | 是 |
| InterruptedIOException | 当前网络连接超时，可能是 VPN 或弱网影响，请稍后重试。 | 是 | 是 |
| UnknownHostException | 当前网络解析不到小票夹服务，请切换网络后重试。 | 是 | 是 |
| ConnectException | 小票夹服务暂时没有响应，请稍后再试；如果一直失败，请联系服务拥有者。 | 是 | 是 |
| 其他 IOException | 暂时连不上小票夹，请稍后再试。 | 是 | 是 |

## 使用规则

1. **优先取后端 error 码**：Android `ExpenseRepository.parseErrorMessage()` 先解析后端 JSON 中的 `error` 字段；已知错误码统一映射为本表文案，未知错误码才回退到 `message`。
2. **不暴露内部信息**：任何情况下不得把堆栈、SQL、文件路径、类名暴露给用户。
3. **不引导重试危险操作**：写操作（确认入账、删除、拒绝）如果已经失败，允许用户重试；但如果用户不确定是否已生效，应先刷新列表确认状态，避免重复操作。
4. **Token 相关错误统一口径**：无论 401 还是 `invalid_token`，统一显示"绑定已失效，请重新绑定账本"，不区分具体原因。
5. **灰度版不显示错误码**：用户界面只显示中文文案，不显示 `invalid_token`、`server_error` 等代码。
