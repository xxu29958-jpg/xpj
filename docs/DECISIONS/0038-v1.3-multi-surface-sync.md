# 0038 v1.3 Multi-Surface Sync — Optimistic Concurrency + Android Offline Outbox + Undo

- Status: accepted
- Date: 2026-05-26
- Decision makers: 项目维护者

## Context and Problem Statement

v1.2.0 之后多端同步行为不一致：

- Backend `expense_service/_ocr.py:_claim_pending_expense_for_ocr` 已用 `UPDATE WHERE updated_at = expected_updated_at` + 409 `expense_changed` 雏形，**但其它 PATCH/DELETE endpoint 没普及**——客户端 stale write 直接覆盖远端。
- Android Room 是 cache，**离线编辑后没有 mutation queue**——上线后用户改动会被默默丢失，或反过来覆盖远端最新。
- `/web` service worker 是 PWA install shell（[[0028]] + Issue #20），`sw.js` 明示 `/api/*` 走 network-only，**没 bgsync / offline mutation**。
- 三端都没有撤销机制——误删 / 误改后只能手动还原。

家庭账本场景多端编辑同一资源理论存在，但 home server + 少数家庭成员 ≠ 协作工具（Notion / Linear）；不需要 CRDT / vector clock / OT。

问题：v1.3 主线"自托管多端同步增强"必须先 codify 同步 invariant，避免后续每个 PR 各发明轮子。

## Decision Drivers

- [`ENGINEERING_RULES`](../rules/ENGINEERING_RULES.md) §0 裁决顺序：数据正确 > 安全边界 > 稳定运行 > 闭环 > 可维护
- §6 服务端是业务真源；同步用 upsert + `(tenant_id, server_id)` 唯一键
- §7 并发写入必须有唯一约束、锁或版本号保护
- §14 "当前阶段不引入 `client_request_id`"——上传/创建幂等由服务端业务键（image hash / idempotent confirm）保证；本 ADR 不解禁
- §0 #5 可维护——拒绝引入 CRDT / vector clock 等大模型

## Considered Options

- **A. 维持现状**——只在 `_ocr.py` 个别场景 409；其它端 last-write-wins，靠用户刷新发现冲突。
- **B. 三层：optimistic concurrency 普及（body `expected_updated_at`）+ Android offline outbox（Room + WorkManager drain）+ 撤销（soft-delete + restore window）**（本 ADR 选）
- **C. 撤销 §14，引入全链路 `client_request_id`**——HTTP header 风格幂等键 + retry dedup。
- **D. CRDT / vector clock / hybrid logical clock**——业界协作工具方案，单服务器单家庭场景过设计。

## Decision Outcome

Chosen option: **B — optimistic concurrency 普及 + Android offline outbox + 撤销**

总不变量是**服务端权威 + 客户端永远不无声覆盖**。三层规则：

- **Optimistic concurrency**：所有 mutate（PATCH / DELETE / 状态机转换）endpoint 在 request body 接受 `expected_updated_at: ISO8601`。Server `UPDATE ... WHERE id, tenant_id, updated_at = expected` 模式；rowcount=0 → 当前行 not_found 返回 404，否则返回 **409 `state_conflict`** + 中文 message + 当前 server 状态。不用 HTTP `If-Match` header：与现有 `_ocr.py` body 字段路径一致，避免 endpoint 一半 header 一半 body 双轨。**不引入 `client_request_id`**——`expected_updated_at` 仅做并发保护，不做请求级 dedup。

- **Android offline outbox**：Room 新增 `pending_mutations` 表（type / target_id / payload / `expected_updated_at` / status / retry_count / created_at）；离线 mutation 先入 outbox + 乐观更新 UI；连上后 WorkManager（[[0030]] 已有）drain。Drain 顺序 = `created_at` 升序；同 target_id 必须串行。409 时 status → conflict，UI surface 给 user 选"覆盖远端"（重发带新 `expected_updated_at`）或"丢弃我的"。`/web` 暂不做 offline mutation（service worker 仍 network-only），v1.4 视需要再开 ADR。

- **撤销**：mutate / delete endpoint 之外加 `POST /api/<resource>/{id}/undo`（与 PATCH/DELETE 对称的 restore）；soft-delete 用 `status='deleted'` + `deleted_at`，5min retention 后 `cleanup_service` 永久删；UI 不可见 soft-deleted 行。客户端三端 undo banner 5s window；window 内点 undo → POST /undo；超时 server cleanup。撤销动作写入 [[0022]] `ledger_audit_logs` `action='undo'`，与其它 audit 共存。

A 被否：现状已知会丢用户改动，违反 §0 数据正确。C 被否：§14 不允许，且 `expected_updated_at` 已能覆盖 OT 场景；D 被否：违反 §0 #5 可维护。

## Consequences

Good:

- §6 / §7 invariant codified；后续 PR 不再发明轮子
- last-write-wins 丢字段语义被 409 拦住——user 显式选择保留哪边
- Android 离线编辑不再被默默覆盖
- 撤销 5s window 覆盖典型误操作

Bad:

- 所有 mutation endpoint schema 都新增 `expected_updated_at` 字段——首版可空（向后兼容旧客户端），后续版本收紧（§6 三步走）
- Android outbox 是新 schema migration——Room 升版本号；旧 Room schema → 新 schema 需可空字段过渡
- `/web` 体验不对称——Android offline 可用，/web 不可；用户文案需明示
- last-write-wins 丢"两端改不同字段"合并语义——家庭场景接受这个 tradeoff，UI 必须 surface

## Confirmation

- Backend contract test：每个 PATCH/DELETE/状态机 endpoint 模拟 stale `expected_updated_at` 必返 409 `state_conflict` + 当前 server 状态
- Android: outbox drain 顺序断言（同 target_id 串行）；409 → conflict UI 状态机
- 撤销 e2e: soft-delete → 5s window 内 undo 恢复；5s 后 cleanup 永久删除
- `ledger_audit_logs` 写入断言：撤销动作有 `action='undo'` 行；不破坏现有 audit query
- `release_audit.py` 加新 audit 检查所有 mutate route 都接 `expected_updated_at`
