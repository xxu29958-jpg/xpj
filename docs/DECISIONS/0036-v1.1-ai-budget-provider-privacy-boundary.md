# 0036 v1.1 AI Budget Provider Privacy Boundary

## Current implementation note

The live provider envelope is intentionally narrower than the original v1.1
draft allowlist. Current production code sends only:

- `month`
- `home_currency`
- `category_breakdown[]` with `category`, `amount_cents`, `count`
- `historical_baseline[]` with `category`, `median_cents`, `p75_cents`
- `income_plan[]` with a generalised `source_type`, `amount_cents`, `pay_day`
- `recurring_total_monthly_cents` + `recurring_active_count` — a coarse
  fixed/recurring-commitment aggregate (two scalars, no rows)

The fixed/recurring summary is intentionally coarse. Recurring items are keyed
by real merchant (`merchant_name`), which is PII, so the builder sends only the
aggregate magnitude — total active monthly commitment and how many active items
— never per-merchant rows, amounts, or names. Paused / archived items are
excluded. The outbound guard fail-closes on either scalar that is not a
non-negative int, so a free-text value can never ride through a scalar slot.

The income `source_type` is free text in storage (a user could type an employer
name there), so the builder generalises every value to a fixed PII-free
allowlist — anything unrecognised collapses to `other` — and the outbound guard
fail-closes on any `source_type` outside that set. The free-text income `label`
is never sent. `amount_cents` is the planned amount and `pay_day` is a
day-of-month (1-31), never a full date.

Income storage can contain both monthly recurring rows and one-time rows tied
to a local `income_month`. The builder filters income rows to the advice month
before constructing `income_plan[]`; `frequency` and `income_month` remain
local-only routing fields and are not sent to external providers.

Every outbound `category` value must be one of `DEFAULT_CATEGORIES`. Existing
ledger rows with legacy aliases are normalized before aggregation; any
remaining custom or polluted category is folded into the default "other"
category before it can reach a provider. The outbound guard also rejects
programmatic `BudgetInputs` payloads whose category values are outside the
default catalog, so provider calls fail closed even if a future caller bypasses
the builder.

Local alias tables may still exist for local interpretation, but
`merchant_summary`, `members`, and per-merchant `fixed_expenses` rows are not
part of the outbound provider payload — only the coarse recurring aggregate
above crosses. Adding any of them back (or making the recurring summary more
granular than the two scalars) requires code, golden-payload tests, and this
ADR to change in the same review.

Live-provider audit rows store only a keyed input hash and call metadata. The
`retention_days` value is snapshotted at write time from
`BUDGET_ADVISOR_AUDIT_RETENTION_DAYS`; changing the environment variable later
does not recalculate old rows. This is intentional so historical audit rows keep
the retention policy that was active when the external call was made. Operators
who need to shorten already-written retention must run the explicit cleanup
path after changing policy, not rely on silent reinterpretation.

## Context and Problem Statement

v1.1 主线是"家庭现金流预算"——本地建立收入 / 固定支出 / 跨月重复 / 本月预计消费 / 历史弹性基线，最终算"本月可自由支配"。AI 在其中只做**解释 + 建议**，不是预算真相来源。

引入 AI 触发隐私问题：原始账本里有真实商户、家庭成员姓名、小票图片、地址、手机号、订单号、交易 ID、备注。把这些发给外部 AI provider 等于把家庭账本外包给模型——违反 [[0017]] "自托管 / 本地优先" 边界，且即使用户日后撤换 provider 也无法收回已上传的数据。

问题：v1.1 接入 AI 之前必须把"AI 能看到什么、不能看到什么、本地保留什么"作为不可逆架构边界**先 codify**，否则任何 AI 集成的 PR 都可能漂移成"把更多数据发上去看效果"。

## Decision Drivers

- 数据最小化 / 用户不应被默认上传任何 PII（GDPR 类原则，但项目无 GDPR 法律义务，仅守工程素养底线）
- AI provider 可换 / 可断网，断网时核心预算仍工作
- 用户能验证"AI 收到的是什么"——开 dev tool 看实际 outbound payload
- 与 [[0028]] 公网边界 / [[0029]] 拆账隐私 / [[0031]] cut-over snapshot 一致：隐私边界一律明示，不靠 reviewer 注意

## Considered Options

- 选项 A：直接把原始数据发 provider，让用户在 UI 选"匿名化"开关
- 选项 B：默认不发任何东西，每个字段单独开关
- 选项 C：**默认最小结构化摘要 + 本地映射表**（推荐）
- 选项 D：彻底不接 AI，只做本地预算建议

## Decision Outcome

Chosen: **C — AI provider 只能看到最小结构化摘要，本地保留真实映射**。

允许上传给 AI 的字段（v1.1 起草时定，新字段加入须更新本 ADR）：
- 分类标签（餐饮 / 交通 / 数码 / etc — 已在代码常量，不算 PII）
- 金额 cents（分类后的聚合或脱敏后的单笔）
- 月份 / 日区间（不到秒）
- 商户**别名占位**（`merchant_001` / `subscription_1` 或泛化类目"超市 / 外卖 / 电商 / 订阅"）
- 收入计划（source_type / amount / pay_day）
- 固定 / 重复 / 预计消费的脱敏摘要
- 历史基线统计（中位数 / P75，**不**逐笔）

明示**不**上传：
- 真实商户名、店铺地址、连锁标识
- 备注字段（默认 off，用户单笔显式勾选才上）
- 小票图片、缩略图、图片 path
- 文件系统路径
- 家庭成员真实姓名（映射 `member_1` / `member_2`）
- 地址 / 手机号 / 邮箱 / 订单号 / 流水号
- 真实数据库交易 ID（换临时 ID）
- 单笔时间戳精确到秒（最多到日）
- ledger / device / token / 任何身份字段

本地保留：`merchant_alias_table` / `member_alias_table` / `transaction_temp_id_table` 三张映射表持久化在 SQLite，**只在本地解释 AI 回答时用**，永不出站。

AI 仅做：(1) 解释结构化摘要 (2) 给建议；**不写预算**——所有预算变更必须用户在 UI 显式确认后才落盘，并写 audit log "采纳了哪个 AI 建议 / 用户手改"。

**Provider 形态**：沿用现有 OCR provider 的 pluggable 模式（见 `app/services/ocr_service.py` 的 `OcrProvider` Protocol + `OCR_PROVIDER` env 切换）。v1.1 加 `BudgetAdvisorProvider` Protocol，但**只两个实现**：

- `EmptyBudgetAdvisor`（**默认**；不调 AI，纯本地规则跑预算。"可填可不填" 的"不填" path）
- `OpenAiCompatBudgetAdvisor`（**统一**本地大模型 + 云端 API。两者都用 OpenAI Chat Completions 协议，prompt + response shape 一致，仅 `base_url` + `api_key` + `model` 三个 env 变量区分）

`OpenAiCompatBudgetAdvisor` 覆盖：

- 本地：ollama (`http://127.0.0.1:11434/v1`) / 自托管 vLLM / llama.cpp server / LM Studio——这些默认就 expose OpenAI-compat endpoint
- 云端：OpenAI / DeepSeek / SiliconFlow / Together / Groq / 国内任何兼容 OpenAI 协议的 provider

非 OpenAI-compat 协议（Anthropic 原生 / Gemini / 百度文心原 API）真要接再加 class——v1.1 默认不带。

`BUDGET_ADVISOR_PROVIDER` env 选 `empty` / `openai_compat`；默认 `empty`。**provider 切换不放宽脱敏边界**——同一份字段允许清单约束所有实现，prompt 模板里能 reference 的字段集是 ADR 列表的子集，新加字段必须先改本 ADR。

## Consequences

Good:
- 用户撤换 provider 也不留数据残骸
- 断网 / provider 故障核心预算不挂
- 任何 PR 增加 AI 调用必须先过本 ADR 字段清单 review
- 自托管定位不被 AI 集成稀释

Bad:
- AI 解释颗粒度只到分类层级——"为什么本月购物预算紧" AI 能答，"为什么这家店扣 580" AI 答不出来（看不到真实商户名）。这是脱敏边界本身的代价，不是 bug

## Confirmation

- outbound payload golden test：AI provider 调用前 dump payload，跟 ADR 允许字段集做 schema 比对，新增字段 fail
- "snake-eating-tail" 测试：模拟 provider 返回的 alias `merchant_001`，本地 映射回真实商户能正确解释给用户
- 断网 fallback test：provider 不可达时预算结果仍由本地规则产生（AI 只是 nullable suggestion）
