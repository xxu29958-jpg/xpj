# 0023 图表库引入政策

日期：2026-05-12

## 背景

早期路线图把"不引入图表库"作为贯穿版本的硬红线。这个约束适合 `v0.4-alpha3 GA` 到 `v0.7`：当时主要工作是稳定身份、账本隔离、家庭协作、Recurring、Rules、Tags 和 Merchant 系统，图表不是主风险。

从 `v0.8` / `v0.9` 开始，预算、趋势报表、商家排行、同比环比、Dashboard 自定义和 PNG 导出会成为主体验。继续长期手写所有图表会把维护成本转移到展示细节上，反而削弱核心记账能力。

图表库不是 UI/UX 美化的替代品。无论是否引入库，统计图都必须明确处理信息层级、颜色语义、深色模式、空状态、加载态、错误态、长标签、窄屏布局和导出预览。

## 决策

小票夹图表库政策（v0.9 起允许引入，已落地 ADR 0025 Android Vico、ADR 0026 Web ECharts）：

图表库只能属于展示层：

- Android 端只能用于 Compose UI 展示，并且必须进入 `android/gradle/libs.versions.toml`。
- Web 端只能用于 `/web` 展示层，不得使用外部 CDN，不得进入公网普通 API，不得影响 `/owner` 运维安全边界。
- 后端仍只返回结构化统计数据，不承担图表渲染，不返回图表库私有格式。

## 约束

引入前必须完成：

- 查官方资料、一手 Maven/npm 元数据、许可证和维护状态。
- 确认不是 alpha、beta、rc、snapshot、停止维护或来源不清的依赖。
- 更新 `docs/rules/REFERENCES.md` 或新增后续 ADR。
- 说明包体积、离线可用性、深色模式、无障碍和导出能力影响。
- 说明失败回退方案：原生 Canvas / SVG 展示必须可恢复。
- 说明统计图 UI/UX 方案：颜色语义、图例、tooltip 或点击态、长商家名处理、空数据、加载中、错误态和移动端窄屏布局。

验证要求：

- Android：`testGrayDebugUnitTest`、`assembleGrayDebug`、`assembleInternalDebug`、`lintGrayDebug`。
- 全局：`scripts\verify_project.ps1`、`scripts\check_text_encoding.ps1`、`git diff --check`。
- UI：覆盖空状态、加载中、错误态、长文本、深色模式和窄屏布局。
- 图表体验：覆盖图例、触摸/鼠标交互、颜色对比度、长标签截断、导出预览和无数据月份。

## 不允许

- 不允许为了图表库引入 React / Vue / Svelte / Node 前端构建体系，除非另有独立架构决策。
- 不允许使用 CDN 拉取图表脚本。
- 不允许图表库进入后端 service、统计口径、预算规则、权限过滤或数据模型。
- 不允许用图表库掩盖分页、聚合、账本隔离或数据正确性缺陷。
- 不允许因为图表展示失败阻塞账单确认、上传、同步、家庭权限或预算规则。

## 后果

该决策把"当前不引入"和"长期绝不引入"分开：

- 当前版本继续保持低依赖和稳定闭环。
- 报表增强阶段可以在受控边界内获得更高质量的可视化能力。
- 如果选型失败，可以回退到原生绘制，不影响核心账本能力。
