# 0037 v1.2 Learning Feedback Dual Tables

- Status: accepted
- Date: 2026-05-25
- Decision makers: 项目维护者

## Context and Problem Statement

V1.2 引入"建议层"（分类自学习、重复候选评分、预算分位数、月报解释、订阅雷达）。这些功能要持续产出建议并消费用户反馈，但**不能污染账本**：账本上的每一行 expense / budget / category_rule 仍然由 [[0022]] 的权限路径写入，不被任何算法直接 mutate。

问题：建议、用户反馈、OCR 原始事实应该写在哪里？最自然的两条歧路都被拒绝——

1. **直接在 `expenses` 上加字段（如 `suggested_category`）** —— 字段会被现有更新路径覆盖，反馈信号无法保留版本史。
2. **复用 `ledger_audit_logs`** —— 那张表是权限审计，混入算法决策会失去其法务可读性。

需要一组独立的 append-only 表承担"系统提议了什么、用户回应了什么、OCR 看到了什么"。

## Decision Drivers

- 账本本体（expenses / budgets / category_rules）不被算法直接 mutate（"建议层避免污染"）
- 算法可任意改版迭代而不影响历史决策可溯源（"评分带来聪明"）
- 用户反馈必须能在算法层被读到并用于降权（"反馈带来个性化"）
- 用户每一次接受 / 拒绝都仍走原有 mutation 路径（"用户确认守住账本"）
- 跨 tenant 零泄漏；个人账本 vs 家庭账本边界跟 [[0022]] 一致
- 表数量克制：不每个建议子类各开一张表

## Considered Options

- **A. 单张 `algorithm_outputs` 表混存决策 + 反馈**
- **B. 反馈 / OCR 事实两张业务 append-only 表 + 决策表（事实 append-only，治理状态可流转）**（本 ADR 选）
- **C. 把每类建议各自单独建表（category_suggestions / duplicate_candidates / budget_suggestions / ...）**
- **D. 完全只用 JSON 列存进 expenses.metadata**

## Decision Outcome

Chosen option: **B. 三张表分两类——事实/反馈是业务 append-only，决策行允许治理状态流转**

总不变量是**账本本体（expenses / budgets / category_rules / ...）不被算法 mutate**。三张表内部规则：

- `algorithm_decisions` —— 算法发出的每一条建议各一行（事实 append-only），但**治理状态可流转**：`status` 在 `active / superseded / withdrawn` 之间转换，`superseded_by_id` 指向新版决策，cleanup 也会删除过期非 `active` 行。决策内容（`decision_type` / `algorithm_version` / `output_payload`）一旦写入不再改。
- `ledger_learning_events` —— 用户对建议的每一次反应各一行；**业务 append-only**，仅在 retention 到期时由 cleanup 删除；`decision_id` 可空（manual_override 路径无对应建议），`before_payload` / `after_payload` 记差异。
- `ocr_facts` —— OCR 每次抽取各一行；**业务 append-only**，仅在 retention 到期时由 cleanup 删除；和 `expenses` 1:N（手动 retry 会写新行），保留 `parsed_*` 结构化字段 + `raw_text`。

A 被否：单表的 `kind = decision | event` 区分会让 query 频繁分支；两者生命周期不同（决策有 supersede，反馈是 immutable）。

C 被否：每类建议各开表会让"列已知的 8 类建议 + 未来 N 类"变成 8+N 张表；schema 演进成本高于 JSON payload 演进。

D 被否：JSON 列没法 `GROUP BY decision_type`，也没法直接索引 `algorithm_version`，失去版本化能力。

## Consequences

Good:

- 算法可改版迭代而不丢历史：旧版决策标 `superseded`，新版指向 `superseded_by_id`
- 一键回滚算法版本：`withdraw_algorithm_version` 把整个 (decision_type, algorithm_version) 翻成 `withdrawn`
- 反馈循环闭合：建议 service 读 `ledger_learning_events` 做 reject 降权
- 三张表 tenant_id 强制；跨 tenant 写 / 跨 tenant supersede 都被 service 层挡

Bad:

- JSON payload 用 `Text` 存（非 SQLite JSON1 列），查询要 `LIKE '%needle%'`；`_count_recent_rejects` 已经踩了一次，append-only 体量增长后是性能债
- 三张表已落地 `retention_days` + `cleanup_expired_learning_tables`；剩余增长风险是 (a) cleanup 没接调度，目前要手动调或靠后续 scheduled task；(b) Owner Console 没暴露三表的体量 / 上次清理时间面板；(c) `algorithm_decisions` 的 `status='active'` 行不被 cleanup 触碰，理论增长上限 = 用户活跃 subject 数 × 算法版本数
- 算法 service 与建议 UI 必须明确：写 `algorithm_decisions` 不等于建议被采纳；UI 渲染 + 用户操作 + 写 `ledger_learning_events` 才闭合
- 现有 OCR 字段（`expenses.raw_text` / `expenses.ocr_draft_fields`）暂时与 `ocr_facts` 双写，未来再做单源迁移
- `decision_type` / `algorithm_version` 已收拢到 `learning_service._algorithm_registry`（`CATEGORY_SUGGESTION` / `DUPLICATE_CANDIDATE` / `BUDGET_SUGGESTION`）；service 端的 `ALGORITHM_VERSION` 现在都从 registry 取，`pending_suggestion_service` 也改用 `<Type>.decision_type` 替代裸字符串。新增算法仍是"加 registry 条目 + 写 service 模块"两步，没自动发现，是有意的扁平。

## Confirmation

- 账本不被污染：`test_learning_service_dual_tables.test_tables_are_isolated_from_ledger` 显式断言写决策 / 事件后 `/api/expenses/pending` 输出不变
- 跨 tenant 写 / supersede 被拒：`test_record_event_rejects_cross_tenant_decision` / `test_supersede_silently_refuses_cross_tenant`
- 反馈降权闭合：`test_suppressed_when_recently_rejected`（分类建议）/ `test_recent_user_reject_applies_penalty`（重复评分）
- 版本回滚：`test_withdraw_flips_active_to_withdrawn` / `test_withdraw_is_tenant_scoped`
- 算法版本化字段不可缺：`algorithm_decisions.algorithm_version` 列 NOT NULL
