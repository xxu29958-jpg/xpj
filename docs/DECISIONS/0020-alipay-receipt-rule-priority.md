# 0020 支付宝账单 OCR 规则优先级

## Status

Accepted.

## Context

支付宝账单详情里同时存在多个金额和多个机构名称。

典型结构：

```text
好想来零食乐园
-17.89
交易成功
订单金额
18.00
碰一下立减
-0.11
支付时间
2026-05-0521:38:13
收单机构
招商银行股份有限公司
```

旧规则把 `订单金额 18.00` 当成实际消费金额，并且把 `收单机构` 里的招商银行误判为商家。

这会导致待确认账单草稿错误：

- 商家错误：招商银行
- 金额错误：18.00
- 分类无法命中真实商户

## Decision

`receipt_parse_service.py` 不再采用“看到一个错例就追加 if”的写法。解析流程改为候选提取和打分：

```text
raw_text
  -> 金额候选：主金额、字段金额、货币符号金额、元金额
  -> 商家候选：交易成功标题、支付成功正文、字段标签、关键词、首行
  -> 时间候选：交易/支付/付款/消费时间高权重，创建/来电/泛时间低权重
  -> 分类候选：商家关键词高权重，全文关键词低权重，同类多证据可累加
  -> 上下文打分：交易成功加权、订单金额降权、优惠/红包降权、机构名降权、广告文案降权
  -> 选择最高分候选
```

新增样本时遵循以下约束：

1. 先把新截图归类为候选来源、噪音过滤、上下文加权或分类提示，不直接散落到主流程。
2. 规则表和权重表集中放在 `backend/app/services/receipt_parse_rules.py`。
3. `backend/app/services/receipt_parse_service.py` 负责解析入口、上下文、证据汇总和最终选择；金额、商家、时间、分类候选生成与校准分别放在 `receipt_parse_amount.py`、`receipt_parse_merchant.py`、`receipt_parse_time.py`、`receipt_parse_category.py`。
4. UI 噪音如状态栏、返回首页、广告、奖励、支付服务入口统一进入过滤/降权集合。
5. 支付方式上下文如 `使用某银行卡支付` 只用于找相邻商家候选，不直接作为商家。
6. 每个真实错例必须变成回归测试，避免后续 OCR 规则互相覆盖。
7. 禁止把某个截图写成单独的长 `if` 分支；如果必须新增判断，它只能作为候选准入、加权、降权或噪音过滤的一部分。

评分模型：

```text
Candidate(value, score, source, dimensions, evidence)
```

- `value` 是候选结果，例如金额分、商家名、UTC 消费时间或分类。
- `score` 是该候选的综合权重。
- `source` 说明候选来自主金额、字段标签、交易成功标题、正文、关键词或时间模式。
- `dimensions` 把评分拆成来源、字段标签、上下文、邻近度、票据场景、结构位置、一致性和噪音八个维度。
- `evidence` 记录加权/降权依据，只给后端测试和调试使用，不暴露到普通 App UI。
- `confidence` 由金额、商家、时间、分类候选分数和 OCR 原文长度共同计算，不再是“有字段就满分”的布尔相加。

维度含义：

| 维度 | 用途 |
| --- | --- |
| source | 候选来自主金额、支付成功标题、字段标签、关键词等来源的基础可靠度。 |
| label | `交易金额`、`支付时间`、`商家` 等字段标签本身的可靠度。 |
| context | `交易成功`、`支付成功`、银行交易提醒等业务上下文加权。 |
| proximity | 候选离交易成功标题、支付方式、金额块越近越可靠。 |
| profile | 支付宝详情、支付成功页、微信支付、银行提醒、出行支付等票据场景先验。 |
| structure | 同一场景里的结构位置，例如主金额块、支付成功正文、出行费用块。 |
| consistency | 同一金额多处出现、商家命中分类、银行提醒首行匹配等互证。 |
| noise | 红包、立减、广告、机构名、状态栏、导航文案等噪音降权。 |

算法分三步：

```text
1. 识别票据场景 profile
2. 生成金额、商家、时间、分类候选并给出维度分
3. 在同类候选之间做互证和冲突校准，再选择最高分
```

场景先验不能直接决定结果，只能调整候选分数。例如支付宝账单详情会提升交易成功附近的签名金额，压低 `订单金额`；出行支付会提升 `高德/打车` 作为商家和交通分类的可信度，但仍必须由候选证据支撑。

新增截图样本也必须先落到同一套模型中：

- 银行 App 动账通知如果没有完整年份，但存在 `动账提醒 / 尾号 / 储蓄账户 / 支出人民币` 等业务上下文，可以生成“无年份消费时间”候选，年份按服务端配置时区的当前年份补齐。该候选仍然要被 `bank_reminder` profile 和支出句子加权后才会胜出。
- 淘宝闪购、美团外卖等付款确认页存在大量商品行、优惠行和底部付款浮层。金额候选要利用结构位置：底部付款浮层里的商户名、支付方式和 `正在付款中` 附近的大金额优先；商品行金额、打包费、配送费、红包、店铺活动、平台红包统一作为商品区或优惠噪音降权。
- 商家候选同理：`店铺活动/券` 这类字段不能当商家；付款浮层里的 `淘宝闪购商户` 可以作为结构化商家候选，但只用于 pending 草稿，用户仍需确认。

支付宝类账单使用以下优先级：

1. 金额优先取 `交易成功 / 支付成功 / 付款成功` 附近的大号签名金额。
2. `订单金额`、优惠、立减、红包、券、奖励金额只作为备选或上下文，不作为实际消费金额优先项。
3. 商家优先取 `交易成功` 上方最近的标题商家。
4. `收单机构`、`清算机构` 里的银行名称不作为商家。
5. 银行名称仍可在银行交易提醒类截图里作为商家或来源，例如 `中国建设银行 + 交易提醒`。
6. `好想来`、`零食`、`小吃` 默认归类为 `餐饮`。
7. 支付宝支付成功页如果真实商家在主金额下方，则把正文里的商家作为候选，广告、森林能量、红包、优惠、评价、打车和奖励文案降权或丢弃。
8. `罗森`、`便利店` 类支付成功页默认归类为 `餐饮`，避免后续广告里的 `高德` 抢成交通。
9. `超市`、`批发`、`商超` 类商家归类为 `购物`，这是泛化分类提示，不为单个商家硬编码。
10. 微信支付详情里的 `交易状态`、`查看账单详情`、`商家名片`、`我的账单` 等是 UI 文案，不作为商家；`使用建设银行储蓄卡支付` 只触发“上一行可能是收款对象”的候选。

## Consequences

- 支付宝账单里的真实付款金额会优先使用 `-17.89` 并保存为 `amount_cents = 1789`。
- 商家会保存为 `好想来零食乐园`，而不是 `招商银行`。
- 支付成功页会保存真实商家 `罗森便利店`，而不是广告区里的 `高德`。
- 当前规则仍然只填 pending 草稿，不会自动确认入账。
- OCR retry 仍不覆盖用户已经手动填写过的字段。
- 旧的建行交易提醒、京东、高德等规则需要继续通过回归测试。
- 规则增长的主要入口是候选生成、候选评分和噪音集合；禁止把每张截图写成独立 if 分支。

## 2026-05-08 真实样本补充

本次新增两类真实错例，继续按“评分维度”处理，而不是在主流程里追加截图专属分支。

### 状态栏数字不能抢金额

样本结构：

```text
08:30
5G
72
支付成功
巴南区卢记牛肉面
¥19.00
摇一摇，有优惠
返回商家
```

旧规则把状态栏里的 `72` 当作金额候选，导致 pending 草稿显示 `¥72.00`。

处理方式：

- 仅当候选来自纯数字行、位于前 5 行、附近有 `4G/5G/WIFI/VPN` 或时间行时，增加 `status_bar_numeric_amount_noise` 降权。
- 如果该行有货币符号或显式正负号，不按状态栏噪音处理。
- 真实金额仍必须通过货币符号、支付成功附近、重复证据或标签证据胜出。

### 短别名附近有更具体商家名时降权

样本结构：

```text
账单详情
闪购
淘宝闪购
-25.68
交易成功
```

旧规则中 `闪购` 和 `淘宝闪购` 都是交易成功上方标题候选，分数打满后因为 `闪购` 行号更靠前而胜出。

处理方式：

- 对支付宝账单详情中的 `success_title` 商家候选，如果短别名附近存在包含该短别名的更长候选，给短别名增加 `short_alias_near_specific_merchant` 降权。
- 这不是淘宝闪购专属规则，也适用于后续“简称 + 完整商家名”相邻出现的账单。
- 不影响微信支付里 `Jack` 这类没有更具体邻近商家名的收款对象。

### 回归要求

以上两个错例均已进入 `backend/tests/test_receipt_parse_scoring.py`：

- `test_status_bar_battery_number_does_not_beat_payment_amount`
- `test_alipay_bill_detail_prefers_specific_merchant_over_short_alias`
